# 모노레포 템플릿

이 저장소는 재사용 가능한 모노레포 구성을 제공하는 템플릿입니다. 효율적인 패키지 관리와 공통 설정을 공유하여 일관된 개발 환경을 구축할 수 있도록 설계했습니다.

## 앱 및 패키지 구조

이 프로젝트는 다음과 같은 디렉토리 구조를 따릅니다:

```bash
/
├─ apps/        # 실제 실행되는 애플리케이션
└─ packages/    # 여러 앱에서 공통으로 사용하는 라이브러리
```

- **`apps/`**: 실제 독립적으로 실행되는 애플리케이션들이 위치합니다. (예: React, Vue, Next.js 앱 등)
- **`packages/`**: 여러 애플리케이션에서 공유하는 패키지들이 위치합니다. (예: config, ui, hooks, utils 등)

## 사용 도구

- **pnpm**: 고성능 및 디스크 효율성이 뛰어난 패키지 매니저로, 모노레포 환경에서 패키지 간 의존성 관리와 연결을 위해 사용합니다.

## 패키지 연결 방식 (pnpm Workspace)

pnpm은 `pnpm-workspace.yaml` 설정 파일을 통해 `apps/`와 `packages/` 내의 각 프로젝트를 하나의 워크스페이스로 관리합니다.

```yaml
packages:
  - "apps/*"
  - "packages/*"
```

- **심볼릭 링크 (Symbolic Links)**: pnpm은 설치 시 워크스페이스 내의 다른 패키지를 참조할 때 `node_modules` 폴더 내에 심볼릭 링크를 생성하여 직접 연결합니다. 이를 통해 실제 파일을 복사하지 않고도 로컬 패키지를 마치 외부 라이브러리처럼 참조할 수 있습니다.
- **`workspace:` 프로토콜**: `package.json` 내에서 워크스페이스 내부 패키지를 의존성으로 추가할 때 다음과 같이 사용합니다:

  ```json
  {
    "dependencies": {
      "@repo/ts-config": "workspace:*"
    }
  }
  ```

  이 방식은 pnpm이 해당 패키지를 원격 레지스트리가 아닌 로컬 워크스페이스 내에서 찾도록 보장하며, 배포(publish) 시에는 자동으로 실제 버전으로 변환됩니다.

  **참고: 패키지 명명 규칙**

  예시로 사용된 `@repo/ts-config`는 **Scope(스코프)**와 **Name(패키지명)**으로 구성된 NPM 패키지 네이밍 규칙을 따릅니다.
  - **`@repo`**: 프로젝트나 조직을 나타내는 Scope입니다. 이를 통해 내부 패키지임을 명확히 구분하고 NPM 레지스트리의 이름 충돌을 방지합니다.
  - **`ts-config`**: 해당 패키지의 기능을 나타내는 실제 이름입니다.

  이 이름은 해당 패키지의 `package.json` 파일에 정의된 `name` 필드에서 유래합니다.

  ```json
  // packages/ts-config/package.json
  {
    "name": "@repo/ts-config",
    "version": "0.0.0",
    "private": true
    // ...
  }
  ```
